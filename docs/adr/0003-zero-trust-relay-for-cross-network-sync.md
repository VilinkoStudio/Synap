# ADR-0003：引入零信任 relay 作为去中心化同步的补充

## 状态

已接受

## 背景

Synap 最初的多设备同步设计偏向纯去中心化：设备之间在能够互相发现和连接时直接同步。这个方向与 Synap 的本地优先、用户掌控数据、尽量减少中心服务依赖的理念一致。

但是纯去中心化同步在现实使用中有明显限制：

- 不同设备经常不在同一个局域网内
- NAT、防火墙、运营商网络会阻断直接连接
- 用户不一定愿意或有能力自己架设 Web 端或服务器
- 一个自建 Web 端通常只能服务单个用户的数据汇合，不适合多人公共使用
- 如果要求两端设备必须同时在线，同步灵活性会明显下降

因此需要在保持本地优先和端到端保密的前提下，引入一个可选的中继节点，让不同网络环境下的客户端也能完成数据流转。

这个中继节点不是数据拥有者，也不是可信同步中心。它只负责转发或暂存必要的密文消息，不应感知笔记内容、语义信息、明文索引或用户知识结构。

## 决策驱动因素

- 继续保持本地优先，笔记明文仍由客户端掌控
- 解决不同局域网、NAT、防火墙导致的设备不可达问题
- 降低普通用户使用多设备同步的门槛
- 支持公共 relay 服务被多个用户共同使用
- relay 服务必须是零信任设计，不能读取笔记内容
- 两个同步设备不必同时在线，relay 可以承担异步流转能力
- relay 的具体同步协议细则应由后续设计实现，不在本 ADR 中展开

## 备选方案

### 方案 A：继续坚持纯去中心化同步

做法：

- 只支持局域网发现、直接连接或用户自行配置的点对点连接
- 不提供公共 relay 服务

优点：

- 架构理念最纯粹
- 没有公共服务器运维成本
- 攻击面较小
- 用户数据流转路径最短

缺点：

- 跨局域网同步体验差
- 依赖设备同时在线
- 普通用户很难稳定配置
- 多设备同步能力会被网络环境严重限制

### 方案 B：提供传统中心化云同步服务

做法：

- 服务器保存用户账号、笔记同步状态和可查询数据
- 客户端与服务器同步，由服务器作为事实上的中心数据源

优点：

- 用户体验接近常见云笔记
- 设备不需要互相发现或同时在线
- 服务端可以更容易做冲突处理、索引和管理

缺点：

- 违背 Synap 本地优先和隐私边界
- 服务器成为高价值明文数据目标
- 需要处理完整账号体系、数据隔离、合规和备份恢复
- 用户必须信任服务端不会读取或滥用数据

### 方案 C：引入零信任 relay 中继

做法：

- 提供一个 HTTP relay 服务，用于客户端之间的密文消息流转
- 客户端仍在本地持有明文数据、密钥和同步状态
- 客户端选择同步对端公钥，并用对端密钥体系完成加密和签名
- relay 服务只处理必要的投递、拉取和基础校验，不读取笔记内容
- relay 作为去中心化同步的补充，而不是替代局域网直连同步

优点：

- 显著改善跨网络同步可用性
- 普通用户不需要自建服务器
- 两端设备不必同时在线
- 公共 relay 可以服务多个用户，因为服务器不汇合明文笔记
- 保留端到端保密和本地优先的核心边界

缺点：

- 系统从纯去中心化变成中心化与去中心化结合
- 需要运维公共 relay 服务
- relay 会引入可用性、滥用防护、配额和消息清理问题
- 异步 relay 同步比在线双向协商更复杂

## 最终决策

采用 **方案 C**：引入零信任 relay 作为 Synap 去中心化同步能力的补充。

明确以下原则：

1. relay 是可选中继能力，不替代局域网直连同步。
2. relay 通过 HTTP 与客户端通信，主要提供跨网络、异步的数据流转通道。
3. relay 服务端不保存、不解析、不索引笔记明文。
4. 客户端负责选择同步对端、加密消息、验证签名、维护本地同步状态。
5. relay 可以做必要的身份、签名、收件人、公钥或投递边界校验，但这些校验不能要求服务端获得明文笔记内容。
6. relay 协议细则、消息格式、状态推进规则和清理策略不在本 ADR 中固定，由后续实现和文档记录。

## 决策说明

本决策承认 Synap 需要在纯去中心化理念和现实可用性之间做取舍。

纯去中心化同步在设备同网、可发现、可连接时非常自然，但它不能很好覆盖移动网络、跨城市设备、NAT、防火墙和设备不同时在线等常见场景。要求用户自己架设服务器也不现实，尤其是 Synap 的目标体验是轻量、直接、低负担。

因此 relay 被设计为一个零信任中继，而不是传统云笔记后端。它的职责是帮助密文消息跨网络流转，不成为笔记数据的权威存储，也不参与明文语义理解。

这意味着服务端可以被公共部署、多人共用，但服务端只能看到投递所需的有限元数据和密文载荷。真正的数据合并、同步状态推进、冲突处理和信任判断仍发生在客户端本地。

## 后果

### 正面影响

- 多设备同步不再强依赖同一局域网
- 客户端不必同时在线也能推进同步
- 普通用户不需要自己架设 Web 端或服务器
- 公共 relay 可以服务多个用户而不汇合明文笔记
- Synap 仍能保持本地优先和端到端保密的核心体验

### 负面影响

- 架构从纯去中心化变为混合式
- relay 服务会成为可用性依赖
- 需要设计滥用防护、配额、消息过期、垃圾清理和观测能力
- 异步同步协议需要处理更多中间状态和失败恢复

### 风险与缓解

- 风险：relay 元数据可能泄露设备交互关系或同步频率  
  缓解：尽量减少必要元数据，避免保存可推导笔记内容的字段，并设计合理的过期清理策略。

- 风险：公共 relay 被滥用为任意消息存储或放大器  
  缓解：增加配额、大小限制、签名校验、过期策略和必要的访问控制。

- 风险：relay 不可用导致用户误以为数据丢失  
  缓解：客户端明确区分本地数据、待发送消息和 relay 投递状态；relay 不作为唯一数据源。

## 需要遵守的落地规则

- relay 服务端不得读取、生成或持久化笔记明文
- relay 服务端不得成为同步事实源，客户端本地状态才是权威
- relay 客户端能力可以放入 `core`，但后台调度、网络条件和平台生命周期由宿主层负责
- relay 协议实现应与 HTTP 适配层分开，便于测试异步同步状态机
- 局域网直连同步仍由 `corenet` 和宿主层提供，不因 relay 引入而废弃
