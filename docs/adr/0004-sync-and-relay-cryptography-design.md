# ADR-0004：明确直连 sync 与 relay 的密码学设计边界

## 状态

已接受

## 背景

Synap 当前已经形成两条不同的同步路径：

1. 直连同步：两个节点同时在线，通过双工字节流直接建立同步会话
2. relay 同步：两个节点不必同时在线，通过 relay 投递和拉取密文包

这两条路径面临的密码学约束并不相同。

直连同步的特点是：

- 双方同时在线
- 可以进行多轮往返握手
- 可以在会话开始时完成身份认证与会话密钥协商
- 更适合“建立一次短期安全通道，然后在通道内传多条同步消息”

relay 同步的特点是：

- 发送时对方可能不在线
- 服务端不能被信任为明文参与者
- 消息需要独立封装、独立验证、独立解密
- 更适合“面向收件人打包一封密文信封，再交给中继服务器暂存或转发”

在实现过程中，我们讨论过几种不同的方案，包括：

- 为直连与 relay 复用同一套密码结构
- 对 relay 维护独立的长期 X25519 身份
- 直接由 Ed25519 身份派生长期 X25519，再把认证与加密完全合并为双重 ECDH 的 signcryption 方案

这些方案各有优势，但复杂度、可解释性和安全属性不同。当前需要一份明确文档，说明 Synap 现在采用什么，不采用什么，以及为什么。

## 决策驱动因素

- 直连同步与 relay 同步必须允许使用不同的密码学结构
- 直连同步优先强调会话建立与在线双向认证
- relay 同步优先强调离线收件、端到端保密、服务端零信任
- 服务端不应持有解密能力，也不应参与明文协商
- 设计应尽量保持可审计、易理解、易测试
- 当前阶段优先保留“显式签名 + 显式定向加密”的结构，而不是过早引入更复杂的 signcryption 方案
- 接收方未来私钥泄露后的历史解密风险，在纯异步单向投递模型下无法被完全消除，不应通过过度承诺前向安全来误导实现

## 备选方案

### 方案 A：直连与 relay 完全复用同一套加密通道

做法：

- 无论是否经由 relay，都试图复用 `CryptoChannel`
- 所有同步都建成类似在线会话的临时通道

优点：

- 结构统一
- 密码学概念更少

缺点：

- relay 场景下对方不在线，无法完成双向会话握手
- 单条异步消息不适合强行抽象成在线通道
- 会把“会话密钥协商”和“离线投递信封”混成一个概念

### 方案 B：为 relay 单独维护长期 X25519 身份，并继续使用显式签名

做法：

- 本地同时维护长期 Ed25519 签名身份与长期 X25519 接收身份
- relay 信封使用发送者 Ed25519 签名
- relay 信封使用接收者 X25519 公钥做定向加密
- 每封消息再生成一次临时 X25519，构成 `ephemeral-static` 加密

优点：

- 职责清楚：签名身份与加密身份分离
- 结构传统，容易解释
- 服务端即使知道收件人标识，也无法获得解密能力
- 接收端可以在不在线的情况下被动收信
- 信封包含显式签名，具备清晰的发送者身份验证能力

缺点：

- 需要额外维护一套长期 X25519 身份
- peer 元数据模型更复杂
- 上层需要分清“谁的签名公钥”和“谁的加密公钥”

### 方案 C：直接从 Ed25519 派生静态 X25519，并改为双重 ECDH signcryption

做法：

- 长期只维护 Ed25519 身份
- 发送和接收时，把 Ed25519 长期身份转换为 X25519 长期 DH 身份
- 每封消息再生成临时 X25519
- 通过长期-长期 ECDH 与临时-长期 ECDH 的组合派生最终会话密钥
- 不再依赖独立显式签名，而让发送者身份隐式参与密钥生成

优点：

- 数据库模型更简单，只围绕长期 Ed25519 身份组织
- 身份绑定进入 KDF，认证和加密融合更深
- 比简单的单次 ECDH 更贴近异步 signcryption

缺点：

- 接收方如果始终只依赖一把长期静态私钥，仍然无法获得完整前向安全
- 不具备当前“显式签名可独立验证”的属性，第三方不可公开验证
- 长期身份同时承担签名与密钥协商职责，单点失陷半径更大
- 协议更难解释，工程边界更不直观
- 当前阶段没有强需求要求放弃显式签名结构

## 最终决策

采用 **分路径密码学设计**：

1. **直连 sync 使用 `CryptoChannel` 会话通道**
2. **relay sync 使用 `sealed_envelope` 离线密文信封**
3. **当前保留“显式签名 + 显式定向加密”的 relay 结构**
4. **当前不采用“Ed25519 派生 X25519 后用双重 ECDH 统一认证与加密”的 signcryption 方案**

更具体地说：

### 对直连 sync

- 使用本地长期 `Ed25519` 身份做在线认证
- 在加密模式下，双方会话内临时生成 `X25519` 材料，协商会话密钥
- 直连同步的安全边界是“先建安全通道，再在通道内运行同步协议”

### 对 relay sync

- relay 服务端不是可信解密方
- 每个消息是一个独立 `sealed_envelope`
- 发送者使用长期 `Ed25519` 身份对信封内容签名
- 接收者使用长期 `X25519` 接收身份解密
- 每个信封独立生成临时 `X25519`，使用 `ephemeral-static` 方式派生对称密钥
- relay 只暂存和转发密文，不持有笔记明文解密能力

### 对 Ed25519 与 X25519 的职责划分

- `Ed25519`：长期身份与签名认证
- `X25519`：
  - 在直连 sync 中，用于在线会话内的密钥协商
  - 在 relay sync 中，用于离线收件解密

### 对未采用的双重 ECDH signcryption 方案

当前明确不采用，原因是：

1. 它并不能解决异步投递中“接收方长期私钥未来泄露后历史消息可回溯解密”的根问题
2. 它会削弱当前“显式签名可独立验证”的性质
3. 它会把长期身份同时绑定到签名和 DH，放大单点失陷影响
4. 当前 relay 需求更重视边界清晰与实现稳定，而不是进一步压缩密钥模型

## 决策说明

这份决策的核心不是“哪种密码学方案绝对最先进”，而是“哪种方案更匹配 Synap 当前的同步结构与实现阶段”。

对于直连同步，`CryptoChannel` 的优势非常明确：双方都在线，可以先做身份校验，再建立一次短期安全通道，然后在通道里跑多轮差异协商。这里没有必要把每条消息都单独做成独立信封。

对于 relay 同步，情况正好相反。对方可能不在线，也不可能要求 relay 服务器参与密钥协商。因此每个同步包都必须是一个可以被独立验证、独立转发、独立解密的密文对象。`sealed_envelope` 恰好匹配这个需求。

我们确实认真评估过“长期只保留 Ed25519，再派生静态 X25519”的方案。它在模型简化上有吸引力，也具备一定的隐式认证优势。但当前阶段，它并没有从根本上改善异步收件最关键的安全短板：只要接收者始终依赖一个长期静态私钥收信，就无法获得完整前向安全。既然如此，就没有必要为了统一长期身份模型而牺牲当前更清楚的职责分离和显式签名结构。

因此，当前阶段的判断是：

- 直连 sync 优先使用会话通道设计
- relay sync 优先使用显式签名与定向加密结合的独立信封设计
- 真正更强的异步前向安全，如果未来需要，应通过可轮换接收密钥、signed prekey 或 one-time prekey 等机制演进，而不是仅靠把长期 Ed25519 身份转换成长期 X25519 身份来宣称已经解决

## 当前落地规则

### 直连 sync

- 使用 `core/src/crypto/channel.rs` 中的 `CryptoChannel`
- 认证身份以长期 `Ed25519` 为准
- 加密通道中的 `X25519` 材料仅服务于该次在线会话
- 直连同步协议运行于已认证的安全字节流之上

### relay sync

- 使用 `core/src/crypto/sealed_envelope.rs` 中的 `sealed_envelope`
- 每个 envelope 必须可在不依赖 relay 明文参与的情况下完成校验与解密
- envelope 中显式携带发送者签名身份信息
- envelope 必须是面向指定接收者的定向加密对象
- relay 服务端可以校验 envelope 格式与签名一致性，但不能获得明文内容

### 设计约束

- 不把 relay 服务器视为可信密码参与者
- 不把“发送者身份混入 KDF”误当作完整前向安全
- 不把当前 relay 设计描述为 signcryption 方案
- 如果未来切换到 prekey 或轮换接收公钥模型，需要新增 ADR，而不是直接改写本 ADR 的结论

## 后果

### 正面影响

- 直连 sync 与 relay sync 的密码学边界清晰
- 上层更容易理解“为什么这里是通道，那里是信封”
- relay 继续保留显式签名语义，便于调试和审计
- 当前实现复杂度保持可控，不引入额外的 prekey 生命周期管理
- 为以后演进到更复杂的异步安全模型保留空间

### 负面影响

- 需要长期维护两种不同的密码学封装思路
- relay 路径仍然需要一套独立长期接收身份
- 异步 relay 仍不具备完整前向安全
- 文档和实现都需要持续强调“显式签名”和“离线定向加密”是不同职责

### 风险与缓解

- 风险：后续开发误以为 relay 与直连只是 HTTP 与 socket 的差别  
  缓解：明确两者在密码学边界上就是不同模型，本 ADR 即为约束文档。

- 风险：误把 Ed25519 转 X25519 的双重 ECDH 方案当作“更先进，所以应立即替换”  
  缓解：本 ADR 已明确该方案的优点与限制，当前阶段不采用。

- 风险：团队成员误以为当前 relay 已经具备完整异步前向安全  
  缓解：明确说明接收方长期私钥未来泄露时，历史密文仍可能被回溯解开。
