# ADR-0001：core 保持平台无关，网络与发现能力外置

## 状态

已接受

## 背景

Synap 的 `core` 需要承载同步协议、身份校验、信任建立、统计记录等核心逻辑，并被多个宿主复用：

- Android：通过 `coreffi` 接入
- 桌面端：Rust 宿主
- Web / Leptos 端：Rust 宿主

同步能力天然会涉及网络传输，但不同平台在网络层的差异非常明显：

- Android 需要面对系统权限、前后台生命周期、网络切换、系统网络 API、mDNS / NSD 等平台能力
- 桌面端更适合直接复用 Rust 网络栈
- Web 端未来很可能需要完全不同的连接方式

如果把具体监听、连接、发现、mDNS 等逻辑直接塞进 `core`，会让 `core` 背负大量平台相关复杂度，削弱其可控性与可复用性。

另一方面，同步真正“最容易出错、最值得统一”的部分，并不是 socket 如何建立，而是：

- 握手流程
- 对端身份验证
- 信任状态流转
- 数据同步协议
- 错误边界
- 同步统计记录

这些逻辑跨平台稳定、风险高、又必须一致，更适合收敛在 `core` 中。

## 决策驱动因素

- `core` 必须尽量保持平台无关
- `core` 内部错误边界需要可控、可测试、可复用
- Android 侧接入不能因为 Rust 直接接管网络层而变得复杂
- mDNS / 监听 / 连接管理属于平台差异较大的能力
- 同步协议本身需要能够复用到多个宿主
- 后续允许 Rust 宿主拥有自己的默认网络实现，但不能污染 `core` 的职责边界

## 备选方案

### 方案 A：`core` 只接收原始读写通道，具体网络层全部外置

做法：

- `core` 只暴露类似 `initiate_sync(read + write)`、`listen_sync(read + write)` 的能力
- 平台层负责监听、建立连接、mDNS、通道生命周期
- Android 通过 `coreffi` 把平台侧通道注入到 `core`

优点：

- `core` 边界最干净
- 平台差异被限制在宿主层
- Android 接入最自然
- 同步协议与网络协议彻底解耦

缺点：

- Rust 宿主如果都要做同步，需要重复实现网络层
- 对只想快速复用同步能力的 Rust 前端来说，接入门槛偏高

### 方案 B：把监听、连接、发现等网络层全部放进 `core`

做法：

- `core` 自己提供 `start_server` / `stop_server` / `connect` / `discover` 之类接口
- 各平台只调用这些高层接口

优点：

- 表面上调用路径更短
- Rust 宿主复用时更“开箱即用”

缺点：

- `core` 会掺入明显的平台相关逻辑
- Android 对接会更复杂，尤其是权限、生命周期、系统网络 API
- mDNS 等能力会让 `core` 职责膨胀
- 错误边界变差，测试与调试复杂度上升

### 方案 C：保持 `core` 平台无关，同时在 `core` 外提供可选的默认网络实现

做法：

- `core` 继续只处理原始读写通道
- 在外层新增可选的 Rust transport / corenet 层，提供 TCP / QUIC / mDNS 等默认实现
- Android 可以继续不用这层，保持平台侧实现
- 桌面 / Web / Rust 宿主可以复用这层

优点：

- 不破坏 `core` 边界
- Rust 宿主不需要重复造轮子
- 平台特殊能力仍然可以按平台实现

缺点：

- 需要维护两层接口
- 架构上比“全塞进 core”多一层抽象

## 最终决策

采用 **方案 C**，并明确以下原则：

1. `core` 的职责是承载平台无关且高风险的同步逻辑。
2. `core` 不直接承担平台相关的网络监听、连接管理、mDNS / NSD、权限与生命周期处理。
3. `core` 对外以“原始读写通道”作为同步入口边界。
4. Android 继续使用平台侧网络实现，并通过 `coreffi` 将读写通道注入 `core`。
5. 若桌面端、Leptos / Web Rust 宿主需要统一网络实现，应在 `core` 外新增可选的 Rust transport 层，而不是把具体网络能力塞回 `core`。

## 决策说明

这个决策的核心不是“Rust 能不能实现网络层”，而是“什么东西应该进入 `core`”。

我们认为：

- 同步协议、信任建立、身份校验、统计记录，是应该强制统一的
- 监听、连接、服务发现、网络权限、系统 API，是应该按平台处理的

这样做的好处是，`core` 会更像一个稳定的协议引擎，而不是一个膨胀的跨平台运行时。

## 后果

### 正面影响

- `core` 保持清晰、稳定、可测试
- Android 侧可以自由使用最适合平台的网络能力
- 同步协议与具体传输方式解耦，未来可以接入 TCP、QUIC 或其他通道
- 后续可以为 Rust 宿主补一个官方默认 transport 层，而不破坏现有边界

### 负面影响

- 纯 Rust 宿主如果没有默认 transport 层，会面临重复实现网络接入的问题
- 调用链相较于“一个函数包办所有网络逻辑”的设计更长
- 平台层与 `core` 之间需要维护一层清晰的 transport 契约

### 需要遵守的落地规则

- 不要为了图省事，把 Android / 平台特有的网络逻辑塞回 `core`
- mDNS、监听端口、连接生命周期管理默认放在平台层或可选 transport 层
- `core` 的服务门面优先暴露协议能力，而不是平台运行时能力
- 若将来增加 `corenet` 或类似模块，应将其视为 `core` 的外层适配器，而不是 `core` 本体

## 后续事项

- 为 Rust 宿主设计一层可选的默认 transport 实现
- 明确 transport 层与 `core` 之间的最小接口契约
- 在后续 ADR 中单独记录默认 transport 层的职责与 API 边界
